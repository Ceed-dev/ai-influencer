## 1. コスト概要

v5.0の運用コストは以下の6カテゴリに分類される。v4.0からの最大の変化は**人件費の削減**と**Claude APIコストの新規追加**である。

| カテゴリ | 内容 | 課金モデル |
|---------|------|-----------|
| コンテンツ制作API | fal.ai (動画生成 + リップシンク) + Fish Audio (TTS) | 従量制 (コンテンツ量に比例) |
| AIエージェント (Claude API) | 戦略Agent / リサーチャー / アナリスト / ツールスペシャリスト / データキュレーター / プランナー | 従量制 (トークン量に比例) |
| インフラ | GCE VM + PostgreSQL + Google Drive | 月額固定 |
| プラットフォームサービス | X Premium + Embedding API | 月額固定 + 従量制 |
| Fish Audio月額 | TTS基本料 (Plus plan) | 月額固定 |
| 人件費 | 監視のみ (v4.0のインターン人件費を削減) | 月額固定 |

**コスト構造の特徴**:
- **変動費の大部分はコンテンツ制作API** — スケールに直接比例
- **Claude APIは変動費だが増加は対数的** — プランナー増設で段階的に増加
- **固定費はインフラのみ** — GCEのスペック変更以外は一定
- **人件費は劇的に減少** — v4.0のインターン4〜12人 → v5.0は監視要員のみ

### ツールスペシャリストによる代替ツール選択時のコスト変動

| ツール入替 | デフォルト単価 | 代替ツール | 代替単価 (推定) | 変動幅 |
|-----------|-------------|----------|---------------|-------|
| 動画生成: Kling → Runway Gen-3 | $0.35/セクション | Runway Gen-3 Alpha | $0.50-0.80/セクション | **+43%〜+129%** |

| TTS: Fish Audio → ElevenLabs | ~$0.001/セクション | ElevenLabs | $0.01-0.03/セクション | **+900%〜+2,900%** |
| Lipsync: fal.ai → Hedra | $0.42/セクション (Pro) | Hedra | $0.30-0.50/セクション | **-29%〜+19%** |

> ツールスペシャリストは品質最適化を主目的とするが、コスト制約もproduction_recipesテーブルに記録されたデータに基づき考慮する。デフォルトレシピからの乖離が大きい場合は、プランナーまたは戦略Agentの承認を必要とする設計とし、コスト暴走を防止する。

## 3. AIエージェント運用コスト (Claude API)

v5.0で新規追加されるコストカテゴリ。AIエージェントがClaudeを呼び出す際のトークン課金。

### Anthropic API料金体系

| モデル | 入力 (MTok) | 出力 (MTok) | キャッシュ書き込み | キャッシュ読み取り |
|--------|-------------|-------------|------------------|------------------|
| Claude Opus 4.6 | $15 | $75 | $18.75 | $1.50 |
| Claude Sonnet 4.5 | $3 | $15 | $3.75 | $0.30 |

### エージェント別トークン消費と日次コスト推定

| エージェント | モデル | 日次トークン消費 (推定) | 日次コスト (推定) |
|------------|--------|---------------------|-----------------|
| 戦略Agent (x1) | Opus 4.6 | 200K-500K | $10-30 |
| リサーチャー (x1-2) | Sonnet 4.5 | 500K-2M x 1-2体 | $5-25 |
| アナリスト (x1) | Sonnet 4.5 | 300K-1M | $3-12 |
| ツールスペシャリスト (x1) | Sonnet 4.5 | 100K-300K | $1-4 |
| データキュレーター (x1) | Sonnet 4.5 | 200K-600K | $2-6 |
| プランナー (xN) | Sonnet 4.5 | 200K-500K x N体 | $2-6 x N |
| ワーカー (コード実行) | なし (LLM不使用) | - | $0 |

**補足**:
- 戦略Agentは日次1回の実行だが、全アカウントのKPIデータを入力に取るため入力トークンが大きい
- リサーチャーは数時間ごとにWeb検索 + 構造化を実行するため、累計トークン消費が多い
- ツールスペシャリストはジョブごとに1回のツール選択判断を行う。トークン消費はコンテンツ本数に比例するが、1回の判断は入力~2K + 出力~500tokensと軽量。50アカウント x 30本/月 = 1,500回/月の判断で約100K-300K tokens/日。加えて、外部ツールのドキュメント読み込み・X投稿の品質分析等の情報収集で追加50K-150K tokens/日を消費
- データキュレーターはキューポーリングで連続実行し、リサーチャーから受け取った生データの構造化・コンポーネント自動生成・重複チェック・品質判定を行う。1件あたり入力~3K + 出力~1K tokensで、日次数百件の処理を想定
- プランナーは担当アカウント数に応じてN体に水平スケールするため、スケールの主要コストドライバー
- ワーカーはLLMを使わないコード実行のため、Claude APIコストは発生しない

### スケール別Claude API月額推定

プランナー1体あたり20〜50アカウントを担当する設計に基づく推定。

| 規模 | アカウント数 | プランナー数 | 日次Claude API | **月額Claude API** |
|------|------------|------------|--------------|-------------------|
| Phase 1 | 50 | 1 | $21-77 | **$630-2,310** |
| Phase 2 | 160 | 2-3 | $31-124 | **$930-3,720** |
| Phase 3 | 340 | 4-7 | $41-164 | **$1,230-4,920** |
| Phase 4 | 1,480 | 15-30 | $61-304 | **$1,830-9,120** |
| Phase 5 | 3,500 | 35-70 | $101-554 | **$3,030-16,620** |

> 上記にはツールスペシャリスト ($1-4/日)、データキュレーター ($2-6/日)、およびX投稿テキスト生成 ($1-95/日、規模依存) のコストを含む。X投稿テキスト生成は規模に比例して増加するが、Claude API全体に占める割合は小さい。

**コスト変動要因**:
- レンジの下限: Prompt Caching最大活用 + Batch API利用時
- レンジの上限: キャッシュなし + リアルタイム処理時
- Prompt Cachingで繰り返しコンテキスト (System Prompt, DBスキーマ, ツール定義) の入力コストを**最大90%削減**可能
- Batch APIで非リアルタイム処理 (リサーチ、分析) を**50%割引**で実行可能

## 4. インフラコスト

v5.0ではGoogle Sheetsベースからの移行に伴い、PostgreSQLホスティングが必要になる。

| コンポーネント | サービス | 月額 |
|-------------|---------|------|
| GCE VM (本番) | e2-standard-4 (4vCPU, 16GB) | ~$120/月 |
| PostgreSQL | Cloud SQL (PostgreSQL 16+ / pgvector) | ~$70/月 |
| Google Drive | Shared Drive (Google Workspace) | 既存契約 ($0追加) |
| Dockerオーバーヘッド | GCE VM上で実行 | **$0** (追加クラウドサービスなし) |
| ドメイン/SSL | 不要 (内部システム) | $0 |

**GCE VMのスペック選定理由**:
- 4vCPU: LangGraphオーケストレーション + MCP Server + パイプラインワーカーの並行処理
- 16GB RAM: Node.jsプロセスプール + LangGraphオーケストレーション (PostgreSQLはCloud SQLで外部管理)
- e2シリーズ: コスト最適化インスタンス、安定したワークロード向け

**Dockerオーバーヘッドについて**:

v5.0ではdocker-composeによるコンテナ化を採用するが、Docker自体はGCE VM上にインストールして実行するため、追加のクラウドサービス費用は発生しない。Cloud Run, ECS, GKE等のコンテナマネージドサービスは使用せず、docker-compose upで直接VMのリソースを利用する設計。コンテナ化のオーバーヘッド (メモリ・CPU) は微小であり、GCEのスペック選定に既に含まれている。

**PostgreSQLホスティング**:

Phase 1からCloud SQL (PostgreSQL) を使用。マネージドサービスにより、自動バックアップ・HA・スケーリングの運用負荷を排除する。Cloud SQLはpgvector拡張をネイティブサポートしている。

| 方式 | コスト | メリット |
|------|--------|---------|
| Cloud SQL (Phase 1から使用) | ~$70/月 | マネージド、自動バックアップ、HA対応、pgvectorネイティブサポート |

> ローカル開発環境では `pgvector/pgvector:pg16` Dockerイメージを使用し、本番Cloud SQLと同等のスキーマで開発・テストを行う。

**開発/本番環境のコスト**:

| 構成 | 追加コスト | 説明 |
|------|----------|------|
| 同一VM上でDocker分離 (docker-compose.dev.yml) | **$0** | 本番とポート・DBを分離するだけ。同一VMのリソースを共有 |
| 別VMで開発環境 (e2-small: 2vCPU, 2GB) | **~$15/月** | 本番に影響を与えない完全分離。ただし開発時のみ起動すれば更に安い |
| 別VMで開発環境 (e2-medium: 2vCPU, 4GB) | **~$25/月** | pgvector含む全機能のテストが可能 |

> Phase 1-2は同一VM上でのDocker分離 ($0追加) で開始し、本番稼働後に必要に応じて別VMに分離する。開発VMは使用時のみ起動することでコストを最小化できる。

## 7. コスト最適化戦略

### 7.1 Prompt Caching (繰り返しコンテキストの最大90%削減)

Claude APIの入力コストの大部分は、System Prompt, DBスキーマ定義, MCPツール定義などの繰り返し部分が占める。Prompt Cachingを活用することで、これらのキャッシュ読み取りコストを通常の入力コストの**10%**に削減できる。

| 対象 | 推定サイズ | キャッシュなし (月額) | キャッシュあり (月額) | 削減率 |
|------|----------|--------------------|--------------------|--------|
| System Prompt + ツール定義 | ~50K tokens | 基準 | 基準の10% | 90% |
| DBスキーマ + ビジネスルール | ~30K tokens | 基準 | 基準の10% | 90% |
| 過去の分析コンテキスト | ~20K tokens | 基準 | 基準の10% | 90% |

### 7.2 Batch API (50%割引、非リアルタイム処理)

リアルタイム性が不要な処理をBatch APIで実行することで50%割引を適用。

| 処理 | リアルタイム性 | Batch API適用 |
|------|-------------|--------------|
| 戦略Agent (日次判断) | 不要 (日次cronトリガー) | **適用可** |
| リサーチャー (市場調査) | 不要 (数時間ごと) | **適用可** |
| アナリスト (パフォーマンス分析) | 不要 (計測完了後) | **適用可** |
| ツールスペシャリスト (レシピ推奨) | 低 (制作ジョブ開始時) | **一部適用可** |
| データキュレーター (データ構造化) | 低 (キューポーリング) | **一部適用可** |
| プランナー (コンテンツ計画) | 低 (日次計画) | **一部適用可** |

> 戦略Agent + リサーチャー + アナリストの全処理をBatch APIで実行した場合、Claude APIコストの約40-60%を対象にできる。

### 7.3 モデル使い分け (Opus = 戦略のみ、Sonnet = それ以外)

v5.0の設計では既にモデル使い分けを組み込んでいる。

| エージェント | モデル | 理由 |
|------------|--------|------|
| 戦略Agent | **Opus 4.6** | ポートフォリオレベルの判断、仮説承認/却下に高い推論能力が必要 |
| リサーチャー | **Sonnet 4.5** | 情報の構造化はSonnetで十分。大量処理のコスト効率を優先 |
| アナリスト | **Sonnet 4.5** | データ分析・パターン認識はSonnetで十分 |
| ツールスペシャリスト | **Sonnet 4.5** | ツール特性の比較・レシピ推奨はSonnetで十分。判断回数がコンテンツ本数に比例するためコスト効率重視 |
| データキュレーター | **Sonnet 4.5** | データ構造化・品質判定はSonnetで十分。連続実行のためコスト効率重視 |
| プランナー | **Sonnet 4.5** | N体にスケールするため、コスト効率が最重要 |

> Opusは戦略Agentのみに限定。Opusの入力コスト ($15/MTok) はSonnet ($3/MTok) の5倍、出力コスト ($75/MTok) は同じく5倍。戦略以外をOpusにすると、Phase 5でClaude APIコストが5倍に膨張する。

### 7.4 fal.aiボリュームディスカウント

fal.aiはプリペイド課金モデルで、大量購入時の交渉余地がある。

| 購入額 | 並行タスク上限 | 備考 |
|--------|-------------|------|
| $100未満 | 5 | 初期テスト向け |
| $100-999 | 10 | Phase 1向け |
| $1,000以上 | 40 | Phase 2以降向け |
| $10,000以上 | 要交渉 | エンタープライズ交渉 |

> Phase 4 (1,480アカウント) 以降は月額$10万超のfal.ai利用が見込まれるため、ボリュームディスカウント交渉を実施すべき。

### 7.5 品質フィルタリング (低品質コンテンツの投稿前除外)

アルゴリズム分析 ([ALGORITHM-CONTENT-VOLUME-ANALYSIS.md](../ALGORITHM-CONTENT-VOLUME-ANALYSIS.md)) の知見に基づき、低品質コンテンツを投稿前に除外することで、無駄なリップシンク費用を削減する。

**フィルタリングの段階**:

| 段階 | タイミング | フィルタ内容 | 削減効果 |
|------|----------|------------|---------|
| スクリプト品質 | 制作前 | プランナーがスクリプトの品質を事前評価 | 制作API費を直接削減 |
| 動画品質 | Kling生成後、Lipsync前 | 生成動画の品質を自動チェック | リップシンク費 ($1.26/セクション) を削減 |
| 最終品質 | Lipsync後、投稿前 | 完成動画の総合品質評価 | 投稿による信頼毀損を防止 |

> 仮に制作前フィルタリングで20%のスクリプトを却下した場合、3,500アカウント規模で**月額~$48,500** (= $242,550 x 20%) のAPI費用を削減可能。

### 収益性の構造的課題

1. **コンテンツ制作APIコストがスケールに比例**: アカウント数 x 30日 x $2.31/本 = 線形増加。収益も線形に増えるはずだが、インプレッション数が計画値の1/10-1/100のため、収益がコストに追いつかない
2. **収益化条件の時間遅延**: プラットフォーム収益化にはフォロワー数・再生数の閾値があり、最短3-6ヶ月。この間は純粋なコスト期間
3. **AI生成コンテンツのエンゲージメント率**: 人間クリエイターの4.1%に対しAI生成は1.8-2.5%。収益化後もCPMが低くなる可能性

### 収益性改善のレバー

| レバー | 効果 | 実現性 |
|--------|------|--------|
| Lipsync Standard版への切替 | コンテンツ制作APIを22%削減 | 高 (品質差の検証が必要) |
| 品質フィルタリングで制作数20%削減 | 制作APIを20%削減 | 高 (v5.0のアナリスト機能で実現) |
| fal.aiボリュームディスカウント | 制作APIを10-20%削減 | 中 (交渉次第) |
| アフィリエイト収益の早期化 | フォロワー閾値不要の案件を獲得 | 中 (営業力次第) |
| 少数精鋭モデル (100アカウント) | 総コストを大幅に圧縮 | 高 (ALGORITHM分析の推奨戦略) |

> **最も効果的な戦略**: ALGORITHM-CONTENT-VOLUME-ANALYSIS.mdが推奨する「少数精鋭モデル」(100アカウント規模) でアカウントあたりのインプレッションを最大化し、成功パターンを確立してから段階的にスケールする。100アカウント規模であれば月間コストは**$5,000-8,000 (¥750K-1,200K)**に収まり、v4.0の人件費削減分だけで賄える。
